# Глоссарий

- A/B‑тестирование (A/B testing) - Трафик делят между вариантами поведения/версии для сравнения метрик и выбора лучшего решения.
- ABAC (Attribute-Based Access Control) - Модель авторизации, где доступ определяется набором атрибутов (пользователь/роль/ресурс/контекст), а не только ролями.
- ADR (Architecture Decision Record) - Короткий документ, фиксирующий одно архитектурное решение: контекст, само решение, альтернативы и последствия (trade-offs).
- Anycast - Сетевой подход, когда один IP анонсируется из разных точек; трафик попадает в "ближайшую" точку по BGP‑маршрутизации; полезно для edge и DDoS‑устойчивости.
- API (Application Programming Interface) - Контракт взаимодействия между системами/компонентами: операции, форматы данных, ошибки, ограничения.
- API Gateway (API‑шлюз) - Пограничный слой, через который проходит трафик к бэкендам; применяет политики безопасности, лимитов, маршрутизации, трансформаций и наблюдаемости.
- API-first - Подход, при котором сначала проектируют и согласуют контракт API (эндпоинты/схемы/события), а затем реализуют код.
- Architecture First (архитектура прежде кода) - Подход: сначала фиксируем границы, контракты, критичные флоу и NFR‑риски (C4/sequence/ADR), затем пишем код.
- ArchUnit - Библиотека архитектурных тестов (Java‑мир) для автоматической проверки правил модульных границ и зависимостей (например, запрет кросс‑модульных импортов).
- Autohealing (автоисцеление) - Автоматическое восстановление компонентов (перезапуск, замена ноды, перенаправление трафика), уменьшающее MTTR.



Backoff - Стратегия ретраев: увеличивать паузу между попытками (часто с jitter), чтобы снизить нагрузку и избежать синхронных всплесков.



Backpressure (обратное давление) - Механизм защиты от перегрузки: downstream сигнализирует upstream замедляться/ограничивать поток, чтобы не накапливались очереди.



BFF (Backend for Frontend) - Клиент‑специфичный фасад/агрегатор (например, отдельно для web и mobile), оптимизирующий API под конкретный UI.



Blast radius (бласт‑радиус) - Масштаб последствий сбоя: какая доля пользователей/функций/регионов затронута инцидентом.



Blue‑green deployment - Стратегия деплоя с двумя окружениями (blue/green) и переключением трафика для минимизации риска и downtime.



BOTEC (Back‑of‑the‑envelope calculations) - "Салфеточные" оценки порядка величин (RPS, трафик, storage, конкурентность) для быстрой проверки реализуемости и узких мест.



Bulkhead (изоляция ресурсов) - Паттерн отказоустойчивости: разделение ресурсов (пулы потоков/соединений/очереди) по классам нагрузки, чтобы сбой одного направления не "утопил" все.



C4‑PlantUML - Библиотека шаблонов PlantUML для построения диаграмм C4 в формате "диаграммы как код".



C4‑модель (C4 model) - Метод описания архитектуры на уровнях детализации: Context → Container → Component (в курсе уровень Code не используется).



Cache stampede - Ситуация, когда много запросов одновременно промахиваются по кэшу (часто из‑за TTL) и обрушивают БД; лечится single‑flight/локами/джиттером TTL.



Cache‑aside - Паттерн кэширования: читаем из кэша → при промахе читаем из БД → пишем в кэш; при записи - обновляем/инвалидируем кэш.



Canary deployment - Постепенный деплоймент новой версии на небольшой процент трафика с наблюдением метрик перед расширением.



CAP‑теорема (CAP theorem) - Рамка компромиссов распределенных систем: при сетевых разделениях нельзя одновременно гарантировать строгую консистентность и доступность.



CDN (Content Delivery Network) - Сеть доставки контента ближе к пользователю, разгружающая origin и уменьшающая задержки на выдаче статики/медиа.



Cell‑based архитектура - Подход изоляции системы на "ячейки" (cells), чтобы уменьшать blast‑radius и проще масштабировать/обслуживать большие системы.



CI/CD - Практики и пайплайны непрерывной интеграции и доставки/деплоя: сборка, тесты, проверки, выкат и откат.



Circuit Breaker - Паттерн отказоустойчивости: при росте ошибок/таймаутов к зависимой системе "размыкает цепь", чтобы защитить сервис и дать зависимости восстановиться.



Component (C4 Component) - Уровень C4, показывающий крупные смысловые компоненты внутри одного контейнера и связи между ними.



Concurrency (конкурентность) - Число одновременных запросов "в работе". Часто оценивают по закону Литтла: Concurrency ≈ RPS × AvgLatency.



Connection draining - Механизм LB: вывод узла из ротации с аккуратным завершением активных соединений/запросов.



Consistent hashing / Rendezvous hashing - Алгоритмы распределения ключей по узлам с минимальными перемещениями при изменении пула; полезны для кешей, шардов, sticky‑маршрутизации.



Container (C4 Container) - Уровень C4, показывающий деплоимые/исполняемые блоки (web/api/worker/db/cache/queue) и их взаимодействия.



Context (C4 Context) - Уровень C4, показывающий систему в окружении: акторы, внешние системы, границы ответственности и интеграции.



CQRS (Command Query Responsibility Segregation) - Разделение путей записи (commands) и чтения (queries), часто с разными моделями/хранилищами для масштабирования и производительности.



DDD (Domain‑Driven Design) - Подход к проектированию, где границы модулей/сервисов определяют по доменным областям и bounded contexts.



DDoS - Атаки на доступность сервиса (L3/L4 floods и L7 HTTP floods); требуют защитных слоев: rate‑limit, WAF, Anycast, scrubbing и т.п.



Deduplication (дедупликация) - Устранение дублей запросов/событий (по idempotency key/event_id и т.п.), особенно важно при ретраях и at‑least‑once доставке.



Degradation (деградация) - Состояние, когда система формально отвечает, но качество ухудшилось (latency, ошибки, урезанные функции); важно отличать от полного простоя.



Deploy (деплой, развертывание) - Процесс доставки и запуска новой версии приложения/конфигурации в среде выполнения.



Distributed tracing (распределенная трассировка) - Наблюдаемость, позволяющая видеть путь одного запроса через несколько сервисов (цепочку спанов) и находить источники задержек/ошибок.



DR (Disaster Recovery) - Подготовка к крупным авариям (падение региона/ЦОДа): планы восстановления, переключения, репликации и тесты сценариев.



Edge burst - Эффект fixed window лимитирования: на границе окна можно кратковременно превысить лимит, если запросы приходят до/после переключения окна.



Egress/Ingress - Исходящий/входящий трафик системы; важен для оценки сетевых лимитов и стоимости.



Elastic throttling - Эластичный троттлинг: разрешать превышение при наличии запаса ресурсов и быстро "схлопывать" при перегрузке.



Elasticity (эластичность) - Способность системы автоматически увеличивать/уменьшать ресурсы под текущий трафик (автоскейлинг вверх/вниз).



End‑to‑end latency - Задержка "глазами пользователя": полный путь от действия клиента до результата через сеть и все компоненты.



EWMA latency - Оценка задержки с экспоненциальным сглаживанием (Exponential Weighted Moving Average), используется в динамической балансировке.



Fail‑closed - Политика отказа защитного компонента (например, лимитера): при сбое стора/правил - блокировать трафик (защита важнее доступности).



Fail‑open - Политика отказа защитного компонента: при сбое стора/правил - пропускать трафик (доступность важнее защиты).



Failover - Аварийное переключение трафика на резервный узел/кластер/регион при отказе основного.



Feature flag (фича‑флаг) - Переключатель поведения/функции без нового деплоя; помогает безопасно выкатывать, быстро отключать и делать canary/A/B.



Fixed Window Counter - Алгоритм лимитирования: счетчик на фиксированное окно времени (INCR + EXPIRE). Дешевый, но дает edge burst на границах окон.



GC pause (пауза сборщика мусора) - Редкие паузы рантайма (например, JVM) часто проявляются как tail latency и ухудшают p99.



GeoDNS - Глобальная балансировка через DNS с учетом географии/латентности/весов; имеет ограничения из‑за кеширования и TTL.



GitOps - Практика управления конфигурацией/инфраструктурой через Git как источник истины: изменения через PR, ревью и автоматическое применение.



Graceful degradation - Плавная деградация: при проблемах система сохраняет ключевые функции, отключая/упрощая второстепенные.



gRPC - RPC‑протокол (обычно поверх HTTP/2) для межсервисных вызовов; часто требует L4/L7 балансировки с учетом долгих коннектов.



GSLB (Global Server Load Balancing) - Глобальная балансировка между регионами/ЦОДами по здоровью/вместимости/латентности (DNS, Anycast, облачные глобальные LB).



Hard throttling - Жесткий троттлинг: при превышении лимита запросы немедленно отклоняются (обычно 429).



Head‑of‑line blocking - Ситуация, когда один "тяжелый" запрос/сообщение блокирует ресурс (поток/коннект/lock), тормозя остальных и ухудшая p95/p99.



Health check - Проверка работоспособности бэкендов балансировщиком/оркестратором, чтобы исключать "плохие" узлы из ротации.



High availability (HA, высокая доступность) - Достижение высокой доли аптайма за счет избыточности, автоматического failover и дисциплины эксплуатации.



Highload (высоконагруженная система) - Система, для которой требования по трафику/данным/пикам выходят за пределы "простого" решения и требуют масштабирования и специальных слоев.



Histogram (гистограмма метрик) - Тип метрики задержек, позволяющий корректно агрегировать распределение по инстансам и считать p95/p99 по кластеру.



Hot key / hot partition - Ключ/партиция, на которую приходится непропорционально много запросов; вызывает перекос нагрузки и рост tail latency.



HTTP 429 Too Many Requests - Типовой ответ при срабатывании rate limiter; часто вместе с Retry‑After и X‑RateLimit‑ заголовками.



IaC (Infrastructure as Code) - Практика описания инфраструктуры декларативно/кодом (репо, ревью, повторяемость), а не ручными настройками.



Idempotency (идемпотентность) - Свойство операции: повторный вызов дает тот же эффект; критично для ретраев, очередей и outbox/event‑delivery.



Ingress - Входной слой/контроллер в кластере (часто L7), который принимает внешний трафик и маршрутизирует его к сервисам.



Incident (инцидент) - Нарушение нормальной работы (ошибки, деградация, простой), требующее реакции и восстановления.



Jitter - Случайная добавка к задержкам (ретраи, TTL), чтобы избежать синхронных всплесков и "стадных" эффектов.



JWT (JSON Web Token) - Формат токена, который шлюз/сервис валидирует (подпись, exp/nbf, aud и т.д.) для аутентификации/авторизации.



Latency (задержка) - Время выполнения действия; важно различать сетевую, очередную, сервисную и end‑to‑end latency.



Layered architecture (слойная архитектура) - Паттерн организации системы по слоям (например, presentation → application → domain → infra) с контролируемыми зависимостями.



L4 load balancing - Балансировка на уровне TCP/UDP (быстро, мало логики). Подходит для долгих соединений и RPC‑трафика.



L7 load balancing - Балансировка на уровне HTTP/gRPC: роутинг по пути/заголовкам, TLS termination, политики; дороже по latency, но "умнее".



LB (Load Balancer) - Компонент, распределяющий запросы между пулом серверов для масштабирования и отказоустойчивости.



Leaky Bucket - Алгоритм лимитирования: запросы "текут" с постоянной скоростью (как из дырявого ведра); сглаживает поток, но плохо пропускает bursts.



Little’s Law (закон Литтла) - Приближение для конкурентности: Concurrency ≈ RPS × AvgLatency; помогает оценить нужные пулы потоков/воркеров.



Load shedding - Сброс части нагрузки (обычно низкого приоритета) при перегрузке, чтобы сохранить работоспособность ядра системы.



Logging (логирование) - Сбор событий и контекста выполнения (желательно структурировано) для расследований, аудита и диагностики.



Maintainability (восстанавливаемость/ремонтопригодность) - Способность быстро возвращать систему в строй после отказа; ключевая метрика - MTTR.



Microservices (микросервисная архитектура) - Набор небольших автономных сервисов с четкими API; дает независимый деплой/масштабирование, но повышает сложность эксплуатации и консистентности данных.



Mirroring / Shadowing - Трафик‑менеджмент: "зеркалирование" запросов на новую версию без влияния на ответ пользователю для безопасной проверки в проде.



Modular Monolith (модульный монолит) - Один деплой, но внутри - строгие модули с границами и публичными фасадами; границы enforced правилами/тестами/ревью.



Monolith (монолит) - Единое приложение и единый деплой, где компоненты сильно связаны; проще старт, но сложнее избирательное масштабирование и эволюция при росте.



Monitoring (мониторинг) - Сбор метрик/состояния, алертинг и дашборды для контроля качества и соблюдения SLO.



MTBF (Mean Time Between Failures) - Среднее время между отказами; метрика частоты сбоев (чем больше, тем лучше).



MTTF (Mean Time To Failure) - Среднее время до отказа; близко к MTBF, часто используют в разных контекстах надежности.



MTTR (Mean Time To Repair/Restore/Resolve) - Среднее время восстановления; ключевой фактор доступности (чем меньше, тем лучше).



Network latency - Часть задержки, обусловленная сетью (RTT, TLS handshake, маршрутизация).



NFR (Non‑Functional Requirements, НФТ) - Нефункциональные требования - характеристики качества системы (доступность, производительность, безопасность и т.п.).



Observability (наблюдаемость) - Способность понимать состояние системы по метрикам, логам и трассировкам.



OAuth2 - Протокол авторизации; на практике часто используется совместно с OIDC и JWT для доступа к API.



OIDC (OpenID Connect) - Надстройка над OAuth2 для аутентификации; часто дает ID‑token (JWT) и стандартные claims.



OpenAPI (Swagger) - Формат спецификации REST API как контракта; помогает API‑first, генерации клиентов/документации и контролю версий.



Outbox pattern (Transactional Outbox) - Паттерн надежной публикации событий: запись бизнес‑данных и события в outbox в одной транзакции; отдельный publisher отправляет в брокер.



Outlier detection - Функция LB/прокси: автоматическое исключение "плохих" инстансов из ротации по ошибкам/латентности.



Overhead (накладные расходы) - Дополнительные задержки/стоимость из‑за сетевых вызовов, сериализации, прокси‑слоев и координации в распределенных системах.



p50 (median) - Перцентиль задержки: 50% запросов быстрее этого значения; "типичная" задержка.



p95 - Перцентиль задержки: 95% запросов быстрее, 5% медленнее; часто базовый ориентир UX/SLO.



p99 - Перцентиль задержки: 99% быстрее, 1% медленнее; отражает "хвост" (tail latency).



p99.9 - Высокий перцентиль задержки: 99.9% быстрее; важен при большом трафике, но требует достаточно данных для стабильных оценок.



Peak load (пиковая нагрузка) - Максимальная нагрузка в характерном окне (суточном/сезонном); часто именно пик определяет архитектуру.



PlantUML - Текстовый инструмент для описания диаграмм (в т.ч. C4 и sequence) как кода; удобен для ревью и CI‑проверок.



PoP (Point of Presence) - Точка присутствия (edge/PoP), где размещают инфраструктуру ближе к пользователям (CDN/Anycast/LB).



Proxy (прокси) - Промежуточный сетевой слой, который маршрутизирует/фильтрует/терминирует соединения и может добавлять политики.



QPS (Queries Per Second) - Количество запросов в секунду (часто - к БД/кэшу).



Quota (квота) - Разрешенный объем операций за период (например, запросы/день на API‑ключ); часто реализуется через rate limiting.



Queueing latency - Часть задержки, которая возникает из‑за ожидания в очередях ресурсов (пул потоков, пул соединений к БД, очередь брокера).



Rate limiting (лимитирование, rate limiter) - Ограничение частоты запросов по субъекту (IP/user/api‑key/route). При превышении - замедление/очередь или отказ (429).



RBAC (Role‑Based Access Control) - Модель авторизации по ролям: права задаются ролями, роль назначается субъекту.



RED metrics - Набор эксплуатационных метрик: Rate (нагрузка), Errors (ошибки), Duration (задержка).



Reliability (надежность) - Вероятность работать без отказов в заданном периоде при заданных условиях; практически - как часто ломается.



Replica (read replica) - Копия БД/хранилища, обычно обслуживает чтения и повышает доступность/масштабируемость чтений.



Replication (репликация) - Копирование данных между узлами (primary→replica) для отказоустойчивости и масштабирования чтений.



Response time (время ответа) - Время для клиента от отправки запроса до получения ответа; включает сеть, очереди, обработку и обратную сеть.



Retry (ретрай) - Повторная попытка операции при временной ошибке; требует таймаутов, backoff+jitter и бюджета ретраев, иначе возможны retry‑штормы.



Retry budget - Ограничение доли/числа ретраев, чтобы ретраи помогали при кратких сбоях, но не превращались в retry‑шторм.



RPS (Requests Per Second) - Запросов в секунду - базовая метрика нагрузки на API/сервисы.



RPC - Удаленный вызов процедур между сервисами; дает сетевые накладные расходы и требует таймаутов/ретраев.



Runbook - Пошаговая инструкция для on‑call/эксплуатации: как диагностировать и восстановить сервис в типовых инцидентах (ускоряет MTTR).



Saga pattern (сага‑паттерн) - Подход к распределенным транзакциям: цепочка локальных транзакций с компенсацией при ошибках (eventual consistency).



Scale‑out (горизонтальное масштабирование) - Рост за счет добавления узлов и распределения нагрузки; требует продуманной работы со state и данными.



Scale‑up (вертикальное масштабирование) - Рост за счет усиления одного узла (CPU/RAM/IO); проще, но имеет "потолок" и большой blast‑radius.



Sequence diagram (диаграмма последовательностей) - Диаграмма, показывающая динамику взаимодействий (кто кого вызывает и в каком порядке) в ключевом флоу.



Service discovery (сервис‑дискавери) - Механизмы обнаружения доступных инстансов сервисов (реестр, xDS и т.п.) в динамической инфраструктуре.



Service Mesh - Инфраструктура межсервисных политик (mTLS, retries, telemetry, traffic policy) обычно через sidecar‑прокси; дополняет API Gateway.



Service time - Время "реальной работы" при обработке запроса (CPU/IO: код, БД, кэш, внешние вызовы), без времени ожидания в очередях.



Sharding (шардирование) - Разбиение данных по узлам по ключу (hash/range/geo) для масштабирования объема и нагрузки.



Single Point of Failure (SPOF) - Компонент, отказ которого выводит из строя весь сервис/систему.



Single‑flight - Техника защиты от stampede: при промахе кэша только один запрос выполняет "дорогую" работу, остальные ждут результат.



SLI (Service Level Indicator) - Измеряемый индикатор качества (например, доля успешных запросов, p95 задержка).



SLA (Service Level Agreement) - Договорный уровень сервиса (обычно внешнее обязательство) с методикой измерения и исключениями.



Sliding Window Counter - Алгоритм лимитирования: приближенная "скользящая" оценка на основе двух окон (текущее+предыдущее); баланс точности и стоимости.



Sliding Window Log - Алгоритм лимитирования: хранение временных меток запросов за окно; точный, но дорогой по памяти/операциям.



Slow‑start - Механизм LB: плавный ввод нового узла в ротацию (постепенный рост доли трафика) для прогрева и снижения флапов.



SLO (Service Level Objective) - Целевое значение для SLI (например, p95≤200мс или availability≥99.9%) в заданном окне.



SRE (Site Reliability Engineering) - Инженерная дисциплина эксплуатации: SLO, автоматизация, снижение MTTR, работа с инцидентами и постмортемами.



Stateful - Компонент хранит существенное состояние локально и сложнее масштабируется горизонтально (пример: write‑master БД).



Stateless - Компонент не хранит долгоживущего состояния локально; состояние вынесено наружу (БД/кэш/объектное хранилище), что упрощает scale‑out.



Saturation (насыщение ресурсов) - Состояние близости к пределам ресурсов (CPU, потоки, коннекты, БД), когда очереди растут и p95/p99 ухудшаются нелинейно.



Sticky sessions - Привязка клиента к одному бэкенду (по cookie/hash). Помогает при stateful‑логике, но ухудшает отказоустойчивость и гибкость.



Strangler Fig Pattern - Стратегия миграции: новая система постепенно "обрастает" вокруг старой, перехватывая трафик и вытесняя старую функциональность.



Soft throttling - Мягкий троттлинг: допускается небольшое превышение лимита (например, +5%), после чего включается ограничение.



Throttling (троттлинг) - Контроль нагрузки (ограничение/замедление) при превышении порога; может быть hard/soft/elastic.



Tail latency - "Хвост" распределения задержек (редкие медленные запросы), который обычно отражается в p99/p99.9 и сильно влияет на UX/SLO.



Timeout - Ограничение времени ожидания ответа; базовый механизм защиты от зависаний и каскадных отказов.



TLS termination - Завершение TLS на LB/Gateway (offload), чтобы разгрузить приложения и централизовать сертификаты/политику.



Throughput (пропускная способность) - Максимальный устойчивый объем работы в единицу времени (RPS, Mbps, msgs/s) при соблюдении SLO.



Token Bucket - Алгоритм лимитирования: "корзина токенов" пополняется с постоянной скоростью, запрос тратит токен; допускает bursts.



TPS (Transactions Per Second) - Транзакций/операций в секунду (часто применяют к БД или бизнес‑транзакциям).



TTL (Time To Live) - Время жизни записи (обычно в кэше); по истечении ключ удаляется автоматически.



UX (User Experience) - Пользовательский опыт; в системном дизайне часто определяется tail latency, ошибками и деградациями.



Waiting time - Часть задержки, когда запрос ждет ресурс (поток, коннект, lock, слот) - главный источник роста tail latency под нагрузкой.



WAF (Web Application Firewall) - Периметровая защита от веб‑атак на уровне HTTP (часто вместе с CDN/LB/Gateway).



Авторизация (AuthZ) - Проверка прав доступа: "что можно делать" с ресурсами (RBAC/ABAC, scopes/claims и т.п.).



Алертинг - Оповещения по метрикам/событиям (SLO violations, рост 5xx, saturation) для сокращения MTTR.



Архитектура - Структура системы: компоненты, границы, данные, протоколы и связи; отвечает на "как устроено" и "почему так".



Атрибуты качества - Категории нефункциональных требований (доступность, надежность, производительность, безопасность, модульность и т.д.).



Аутентификация взаимная (mTLS) - Схема TLS, где и клиент, и сервер предъявляют сертификаты; используется для zero‑trust внутри контура.



Аутентификация (AuthN) - Подтверждение личности: "кто ты" (OAuth2/OIDC, JWT, mTLS, API‑keys и т.п.).



База данных (БД) - Хранилище состояния системы; в дизайне учитывают модель данных, транзакционность, репликацию, шардинг, SLA/латентность.



Балансировка нагрузки - Распределение входящего трафика по пулу узлов для избегания перегрузки и повышения доступности.



Безопасность - Набор мер защиты системы: аутентификация/авторизация, шифрование, защита от DDoS, принцип наименьших привилегий.



Брокер сообщений - Инфраструктурный компонент для асинхронного обмена сообщениями/событиями (очереди/стримы) и выравнивания пиков.



Девятки доступности - Обозначение уровней availability (99.9%, 99.99% и т.д.), где каждая дополнительная "9" резко снижает допустимый простой и удорожает решение.



Доступность (Availability, A) - Доля времени, когда сервис доступен и корректно обслуживает запросы. Формула: A = (T − downtime)/T, приближенно A ≈ MTBF/(MTBF+MTTR).



Каскадный отказ - Цепная реакция: сбой/замедление компонента приводит к перегрузке других и усилению инцидента по всей системе.



Компонент - Строительный блок системы (сервис, БД, кэш, очередь, LB, CDN и т.д.) с четкой ролью и интерфейсом.



Компромисс (trade‑off) - Осознанный выбор между конкурирующими целями (например, простота vs масштабирование, консистентность vs доступность).



Консенсус‑алгоритм - Протокол согласования состояния/выбора лидера в кластере (важен для отказоустойчивых stateful‑систем).



Консистентность данных - Свойство согласованности чтений/записей. Часто выбирают компромисс: strong consistency vs eventual consistency.



Контракт - Договоренность о взаимодействии (API/события): форматы, версии, ошибки, ограничения и ожидания по SLO.



Кэш - Быстрый слой хранения (часто in‑memory, например Redis) для уменьшения latency и снятия нагрузки с БД.



Масштабируемость - Способность системы выдерживать рост нагрузки (RPS/данных/географии/команд) без нарушения SLO при приемлемой стоимости.



Модульность - Способность разбивать систему на части с ясными границами и интерфейсами, чтобы снижать связанность и упрощать развитие.



Отказоустойчивость - Способность системы сохранять работоспособность при сбоях компонентов (избыточность, failover, graceful degradation, защитные паттерны).



Очередь - Инфраструктурный механизм буферизации задач/сообщений для асинхронной обработки и сглаживания пиков.



Перцентили задержки - Статистика распределения latency (p50/p95/p99/…), важна для оценки tail latency и SLO.



Производительность - Способность системы быстро и эффективно выполнять операции при заданной нагрузке (latency, throughput, saturation).



Ретрай‑шторм - Сценарий, когда ретраи многократно увеличивают нагрузку и ухудшают ситуацию (больше таймаутов → больше ретраев → еще хуже).



Сервис - Исполняемый компонент, предоставляющий API/контракт и реализующий часть бизнес‑функций; может быть частью монолита или отдельным микросервисом.



Система распределенная - Система, компоненты которой работают на разных узлах/в регионах и взаимодействуют по сети; требует учета частичных отказов и консистентности.



Сопровождаемость - Насколько легко поддерживать и развивать систему: операбельность (observability/runbooks), простота, изменяемость; напрямую влияет на MTTR.



Функциональные требования - Требования, описывающие "что делает система" (операции/функции/сценарии).



Флап - это быстро повторяющееся "прыгание" состояния компонента туда-сюда: то "здоров", то "не здоров", то "доступен", то "недоступен".
