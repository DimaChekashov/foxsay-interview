# Фабричный метод (Factory Method)

## Что это
Паттерн проектирования который делегирует создание объектов
специальному методу вместо прямого вызова `new`.

**Главная идея:** код не знает какой конкретный класс создаётся —
он просто просит фабрику дать нужный объект.

## Когда использовать
- Тип объекта известен только в рантайме
- Нужно легко добавлять новые типы без изменения существующего кода
- Хочешь убрать растущий if/else при создании объектов
- Нужна подмена реализации в тестах

## Структура
```
Creator (фабрика)
  └── factoryMethod() → Product

ConcreteCreatorA → ConcreteProductA
ConcreteCreatorB → ConcreteProductB
```

## Пример — система оплаты
```js
// Продукты
class StripePayment {
  pay(amount) { console.log(`Stripe: ${amount}₽`) }
}
class PaypalPayment {
  pay(amount) { console.log(`PayPal: ${amount}₽`) }
}

// Фабрика
class PaymentFactory {
  static create(type) {
    const map = {
      stripe: StripePayment,
      paypal: PaypalPayment,
    }
    const Payment = map[type]
    if (!Payment) throw new Error(`Неизвестный тип: ${type}`)
    return new Payment()
  }
}

// Использование
const payment = PaymentFactory.create('stripe')
payment.pay(1500) // Stripe: 1500₽
```

## Плюсы и минусы

| ✅ Плюсы | ❌ Минусы |
|---------|---------|
| Новые типы без правки старого кода | Больше классов в проекте |
| Легко мокать в тестах | Избыточен для простых случаев |
| Один принцип ответственности | |

## Принцип который соблюдает
**Open/Closed** — открыт для расширения, закрыт для изменения.
Добавляешь новый тип → только фабрика меняется.

## В реальных проектах
- Фабрика хранилищ (S3 / локально / GCS)
- Фабрика уведомлений (Email / SMS / Push)
- Фабрика логгеров (Winston / Console / File)
- Фабрика транспорта (REST / gRPC / GraphQL)

## Отличие от простой Фабрики
- **Фабрика** — отдельный класс с реестром типов (проще)
- **Фабричный метод** — метод в базовом классе,
  подклассы переопределяют (гибче, но сложнее)
